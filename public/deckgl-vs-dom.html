Deck.gl vs DOM Markers — GeoMap.tsx

Por qué 18 763 puntos necesitan la GPU, no el DOM

El problema fundamental

Google Maps crea un nodo DOM por cada marker. El DOM es una estructura de árbol que el browser mantiene en memoria, recalcula estilos, reposiciona en cada scroll/zoom, y envía al compositor. Con 18 763 markers, eso son 18 763 elementos HTML activos simultáneamente.

Deck.gl trabaja de forma completamente distinta: convierte todos los puntos a un buffer de bytes en la GPU y los dibuja en un único drawArrays(). La GPU está diseñada exactamente para esto — procesar millones de vértices en paralelo a 60 fps.

Cómo Deck.gl usa la GPU — pipeline WebGL
✓ ScatterplotLayer GeoMap.tsx : 68

Deck.gl empaqueta todos los puntos en un ArrayBuffer tipado y lo sube a la VRAM. A partir de ahí, la GPU opera completamente autónoma — la CPU solo manda la señal de dibujo.

■ posición   ■ color RGBA   ■ radio  — los 18 763 puntos = ~1.3 MB contiguos en VRAM
1 — CPU sube datos a VRAM (una vez)
Deck.gl llama gl.bufferData() con el Float32Array de todos los puntos. Ocurre solo cuando cambian los datos, no en cada frame.
2 — Vertex Shader (GPU, 1 instancia por punto)
El shader lee lon, lat del buffer, aplica la proyección Mercator y calcula la posición en pantalla. Los 18 763 puntos se procesan en paralelo por miles de cores de GPU.
3 — Rasterización
La GPU convierte cada punto (círculo) a píxeles, interpolando coordenadas. Sin paso por CPU.
4 — Fragment Shader (GPU, 1 instancia por píxel)
Lee R, G, B, A del buffer y pinta el píxel. Los puntos con territorio tienen color asignado; los sin asignar reciben [160, 160, 160, 180] (gris).
5 — 1 draw call → frame completo en <16ms
gl.drawArrays(GL.POINTS, 0, 18763) — una sola instrucción dibuja todo el mapa. La CPU queda libre para responder a eventos.
El código — ScatterplotLayer en GeoMap.tsx
GeoMap.tsx : 68

Deck.gl abstrae toda la API WebGL. Tú defines los atributos; Deck.gl compila los shaders, gestiona los buffers y emite el draw call.

const scatterLayer = useMemo(
  () =>
    new ScatterplotLayer<SalesPoint>({
      id: "sales-points",
      data: visiblePoints,                        // JS array → Float32 buffer en GPU

      getPosition: (d) => [d.longitud, d.latitud], // lon/lat → vec2 en vertex shader

      getFillColor: (d) => {
        const color = territoryColorMap.get(d.id)
        return color
          ? [...color, 220]          // color de territorio asignado
          : [160, 160, 160, 180]    // gris por defecto
      },

      radiusMinPixels: 3,              // siempre visible aunque se aleje
      radiusMaxPixels: 12,             // no crece infinito al acercarse
      pickable: true,                  // GPU picking — ray vs buffer
      autoHighlight: true,            // shader cambia color en hover

      updateTriggers: {
        getFillColor: [territoryColorMap],  // recalcula buffer solo si cambia el mapa
      },
    }),
  [visiblePoints, territoryColorMap, onPointClick]
)

El useMemo garantiza que Deck.gl no reconstruya el buffer de GPU en cada render de React — solo cuando cambian visiblePoints o territoryColorMap.

Cómo se cambia el color de un punto
GeoMap.tsx : 59

Cuando se generan territorios, cada punto recibe el color de su vendedor. Con DOM esto requiere element.style.background = color para cada nodo. Con WebGL, Deck.gl reconstruye el buffer de atributos de color y lo re-sube a la GPU — una operación de memoria, no de DOM.

// Construye un Map: pointId → [R, G, B]
const territoryColorMap = useMemo(() => {
  const map = new Map<string, [number, number, number]>()
  territories.forEach((t) => {
    const rgb = hexToRgb(t.color)
    t.puntosVenta.forEach((sp) => map.set(sp.id, rgb))
  })
  return map
}, [territories])

// En el shader accessor — O(1) lookup por punto
getFillColor: (d) => {
  const color = territoryColorMap.get(d.id)
  return color ? [...color, 220] : [160, 160, 160, 180]
}

El updateTriggers: { getFillColor: [territoryColorMap] } le indica a Deck.gl que solo reconstruya el buffer de colores cuando el Map cambie — no en cada render.
