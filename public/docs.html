1Estrategia para 18 000+ puntos

Renderizado GPU con Deck.gl

Los 18 763 puntos se renderizan íntegramente en la GPU mediante Deck.gl ScatterplotLayer. El hilo principal nunca itera sobre los puntos para dibujarlos; el navegador solo ejecuta un draw call por frame, lo que garantiza 60 fps incluso con todos los puntos visibles.

Procesamiento off-thread con Web Worker

El algoritmo de asignación de territorios corre en un Web Worker dedicado (workers/geo.worker.ts). El hilo principal envía los datos y recibe el resultado; la UI nunca se congela durante el cálculo.

Filtro espacial con R-tree (rbush)

Para el filtro de radio, se construye un índice R-tree con todos los puntos. La búsqueda por bounding box es O(log n); luego se aplica distancia exacta con Turf.js solo sobre los candidatos del bbox. 

// Filtro de radio — geo.worker.ts
const tree = new RBush()
tree.load(points)                         // carga en bulk O(n log n)

const candidates = tree.search(bbox)    // O(log n) — candidatos
const result = candidates.filter(p =>
  turf.distance(center, p) <= radiusKm   // distancia exacta
)

Asignación de territorios — Greedy Nearest Neighbor con capacidad fija

Se eligen k semillas aleatorias de puntos reales del dataset (no centroides abstractos). Luego se computan todos los pares punto↔semilla, se ordenan por distancia y se asignan greedily: un punto va al vendedor más cercano que aún tenga cupo. Esto garantiza exactamente N puntos por vendedor sin solapamiento.

¿Por qué no K-means estándar? K-means no tiene restricción de capacidad — un cluster puede tener 5 000 puntos y otro 20. El algoritmo greedy con capacidad fija resuelve exactamente el requerimiento: N clientes por vendedor.

Cache HTTP de puntos

El endpoint /api/points incluye Cache-Control: s-maxage=300. Los puntos no cambian entre requests; cachearlos 5 minutos elimina la carga a la base de datos en recargas y navegación entre páginas.

3Escalabilidad a 500 000 puntos

Con 500k puntos el enfoque actual falla en dos lugares: la descarga inicial (~50 MB de JSON) y el Web Worker (18k × 500k pares = 9 000M operaciones para el greedy). A continuación la estrategia de escala por capa.

DB Clustering en PostgreSQL + PostGIS

Mover el K-means al servidor con ST_ClusterDBSCAN o ST_ClusterKMeans. El índice GiST espacial hace las búsquedas por proximidad en O(log n) en el servidor. El cliente solo recibe el resultado ya agrupado.

API Carga por viewport (lazy tiles)

En lugar de descargar todos los puntos, el cliente envía el bounding box visible del mapa y la API devuelve solo los puntos dentro del viewport. Al hacer zoom, se solicitan más puntos con mayor densidad.

Mapa Vector Tiles (MVT)

Servir los puntos como Mapbox Vector Tiles desde PostGIS (ST_AsMVT). Deck.gl los consume con MVTLayer. El servidor renderiza y agrupa por zoom level — el cliente solo descarga los tiles visibles.

Cache Redis + pre-cómputo

Los territorios calculados se guardan en Redis con TTL largo. Si los mismos parámetros (vendedores + N) ya fueron calculados, se devuelve el resultado cacheado sin correr el algoritmo de nuevo.

Worker Worker Pool + SharedArrayBuffer

Para el greedy en cliente, usar múltiples workers en paralelo con SharedArrayBuffer para el array de asignaciones. Cada worker procesa un subconjunto de pares. Reduce el tiempo proporcional al número de cores disponibles.

Render Clustering visual (Supercluster)

A niveles de zoom bajo, mostrar clusters de puntos con IconLayer (número de puntos por cluster). Solo al hacer zoom se revelan los puntos individuales. Deck.gl tiene soporte nativo para esto.

Flujo propuesto para 500k puntos

  • Zoom bajo (ciudad completa) Supercluster devuelve ~200 clusters con conteo. Deck.gl renderiza 200 iconos, no 500k puntos.
  • Zoom medio (distrito) MVT tiles cargan los puntos del viewport actual (~5 000 puntos visibles). R-tree local para interacción.
  • Zoom alto (calle) Puntos individuales con tooltip y click. La descarga es mínima (<500 puntos visibles).
  • Asignación de territorios ST_ClusterKMeans en PostgreSQL con worker de Node.js dedicado (no serverless). El resultado se guarda y el cliente solo descarga los polígonos finales.
